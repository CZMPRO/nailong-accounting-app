# 注释检查（comments-check）

本技能只做**注释质量检查与报告**，默认**不**批量改业务逻辑。  
若用户明确要求「按报告补注释」，可在报告后征得同意再改对应文件。

---

## 检查目标（三条，缺一不可）

### 1. 注释是否缺少（数量 / 覆盖）

**总原则：** 每个函数、以及每一段重要核心逻辑，都应有**详细中文注释**。

**数量参考（硬指标，用于估算）：**

- 在有效代码区域内，大约 **10 行里约 3 行是注释、7 行是代码正文**  
  → 注释行占比大约 **≥ 30%**（按「注释行数 /（注释行 + 代码行）」粗算，空行可不计或单独注明）
- **每个导出函数 / 重要函数** 至少具备：
  - 函数职责说明（做什么、给谁用）
  - 关键参数/返回值含义（若有）
  - 易踩坑点（若有副作用、写盘、异步、边界）
- **核心业务块**（初始化账本、迁移、统计汇总、IPC 处理、存盘、导入导出等）在逻辑转折处应有行内或块注释，不能「大段黑盒代码零说明」

**什么算注释：**

- `//` 单行注释  
- `/* */` 或 `/** */` 块注释  
- 不含：纯分隔线无文字、只写 `// TODO` 而无说明、被注释掉的大段废代码（废代码应标为「死代码」问题，不计入优质注释）

**什么算「重要核心代码」（优先查）：**

- `electron/database.ts`、`electron/main.ts`、`electron/preload.ts`  
- 统计、导入导出、分类种子与迁移  
- 复杂条件/循环、金额与日期处理、状态同步  
- 用户点名的文件或本次 `git diff` 改动文件  

**默认降低优先级（除非用户点名）：**

- 纯展示型 JSX 里重复的简单绑定  
- 自动生成物、`node_modules`、`dist`、`release`  
- 测试文件里已足够自解释的 `it('中文描述')`（仍建议对复杂 setup 有注释）

### 2. 注释是否与代码匹配（正确性）

逐段对照「注释说的」和「代码做的」：

| 问题类型 | 例子 |
|----------|------|
| 过时 | 注释写 SQLite，代码已是 JSON 文件 |
| 说反 | 注释写「仅支出」，代码已支持 income |
| 夸大/缩水 | 注释写「会校验权限」，代码无校验 |
| 张冠李戴 | 注释描述的是别的函数逻辑 |
| 参数不符 | 注释参数名/单位/默认值与实现不一致 |

发现不匹配 → 记为 **「注释与代码不符」**，并引用文件与行号；**以代码为准**描述真实行为，再建议怎么改注释（或若代码才是错的，单独标「可能是代码 bug」，不擅自大改业务）。

### 3. 是否符合小白视角（可读性）

面向**编程零基础**读者：

**合格注释：**

- 简体中文为主  
- 先说「人话结果」，必要时再补术语  
- 用生活比喻解释抽象概念（如：账本文件像自动保存的 Excel）  
- 说明「为什么」而不只是把代码翻译成中文（避免 `// 把 i 加 1`）

**不合格注释：**

- 全英文堆砌、无中文说明（专有名词可保留英文，但句子要中文）  
- 只有术语缩写、不解释副作用（写盘、删数据、IPC）  
- 语气含糊：`// 处理一下`、`// 临时`、`// fix`  
- 对危险操作无警告（清空流水、恢复出厂、覆盖文件）

## 执行步骤

### 1. 确定检查范围

按优先级：

1. 用户点名的文件/目录  
2. 本次改动：`git status` / `git diff`  
3. 默认核心：`electron/database.ts`、`electron/main.ts`、`electron/preload.ts`，以及 `src` 下复杂页面逻辑（克制抽样，一次不宜上百文件）

向用户用一句话说明：**本次检查哪些文件**。

### 2. 阅读代码

- 用 Read/Grep 阅读目标文件，不要只看文件名就打分  
- 按「函数 / 代码块」为单位做笔记  

### 3. 量化注释密度（每个主要文件）

对每个文件给出近似统计：

- 代码行数（非空、非纯注释）  
- 注释行数  
- 注释占比 ≈ 注释行 /（注释行 + 代码行）  
- 是否达到约 **30%** 参考线  
- 哪些函数完全无说明  

统计允许人工估算 + 工具辅助，需在报告中标明「约数」。

### 4. 匹配性与小白视角

- 列出不匹配注释（文件、行号、注释摘要、代码实际行为）  
- 列出小白不友好注释（过于术语化、无意义翻译、缺风险提示）  

### 5. 输出中文报告（必须）

```markdown
## 注释检查结论
- 总体：通过 / 部分通过 / 未通过（一句话）

## 检查范围
- 文件列表

## 指标 1：注释是否充足（约 30%）
| 文件 | 约占比 | 是否达标 | 缺注释的函数/代码块 |
|------|--------|----------|---------------------|
| ... | ...% | 是/否 | ... |

## 指标 2：注释与代码是否匹配
- （无问题写「未发现明显不符」）
- 问题条目：位置 + 注释原意 + 代码实际 + 建议

## 指标 3：小白可读性
- 做得好的例子（1～3 个，可选）
- 需改进的条目：位置 + 原因 + 改写示例（大白话）

## 优先整改清单（按紧急度排序）
1. ...
2. ...

## 说明
- 未检查的范围
- 是否已按用户要求直接改文件（默认否）
```

### 6. 是否自动改注释

- **默认只报告不修改**  
- 用户说「按报告补注释 / 帮我改好」时，再 Edit 补中文注释；保持与代码一致，不写假注释凑 30%  
- 禁止为刷占比而粘贴无信息注释  
- 不顺手大改业务逻辑

## 与其它技能边界

| 技能 | 关系 |
|------|------|
| `/unit-test` | 测行为对不对；本技能测「说明写没写好」 |
| `/git-save` | 本技能不自动提交 |
| `/run-app` / `/rebuild-app` | 无关启动与打包 |

---

## 沟通与安全

- 全程简体中文；工具 `description` 用中文  
- 结论 → 依据（文件/行号）→ 怎么改  
- 不编造行号；读不到的文件不要硬评  
- 不删除用户数据、不强制推送  

## 成功标准

- 三条指标都有明确结论与例证  
- 范围说清楚  
- 给出可执行的优先整改清单  
- 用户未要求时不批量改代码
