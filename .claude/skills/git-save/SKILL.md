# Git 存档并推送到远程

本技能只负责：**看清改动 → 用中文提交说明存档 → 推到远程**。  
不做功能开发、不改业务逻辑；不创建 GitHub Release、不上传安装包（那是 Releases 网页/API 的事）。

## 目标仓库约定（奶龙记账）

- 工作目录：项目根目录（有 `.git` 与 `package.json` 的 `bookkeeping/`）
- 主分支通常为：`main`
- 远程通常为：`origin`（如 `https://github.com/CZMPRO/nailong-accounting-app.git`）
- **署名**：仅使用用户本机 Git 配置的个人署名；**禁止**添加 `Co-Authored-By: Claude` 或任何 AI 联合署名
- **语言**：提交说明、对用户汇报、工具 `description` 一律简体中文

## 执行步骤（严格按顺序）

### 1. 摸清现状（先看再动）

并行收集：

```bash
git status
git diff
git diff --staged
git log -5 --oneline
git rev-parse --abbrev-ref HEAD
git remote -v
```

用大白话向用户概括：改了哪些文件、是未跟踪还是已修改、当前分支、是否领先/落后远程。

### 2. 决定「存什么」

- **默认**：把本次相关的业务/文档/技能改动加入存档。
- **不要默认加入**（除非用户明确要求）：
  - `node_modules/`、`dist/`、`dist-electron/`、`release/`（安装包体积大，且通常已在 `.gitignore`）
  - 含密钥的本地文件、`.env`、本机 token
  - 纯本机权限/日志：如 `.claude/settings.local.json`、`.claude/states/*.log`
  - 用户未要求时的 `.claude/settings.json`（可询问是否纳入）
- 若工作区**干净**且相对远程无无提交：说明「没有新改动可存档」；若仅本地有未推送提交，可只执行推送（见第 5 步）。
- 若有**敏感信息**疑虑：先停下来问用户，不要推送。

### 3. 起草提交说明（先给用户看再提交）

- 用**一句中文**说清「做了什么、为什么」（可多行正文补充）。
- 风格参考历史：`feat:` / `fix:` / `docs:` 等前缀 + 中文描述（与仓库习惯一致即可）。
- **先把拟用的提交说明展示给用户**；用户已明确说「直接存档/提交并推送/按你写的来」时可不再二次确认文案。
- 用户若改写了说明，以用户的为准。

### 4. 暂存并提交

```bash
# 按需添加文件（示例：具体路径按 status 结果填写，避免无脑 git add .）
git add <文件或目录>

git commit -m "$(cat <<'EOF'
<中文提交说明>

EOF
)"
```

注意：

- 优先**按路径 add**，避免把不该上传的文件塞进去；仅在改动明确且全是可提交内容时，才可用范围更广的 add。
- 提交成功后运行 `git status` 确认。
- 若存在 pre-commit 钩子失败：**不要**使用 `--no-verify` 绕过；把错误原文贴出并用大白话解释，等用户决定。
- 本仓库若无提交钩子，则正常提交即可。

### 5. 推送到远程

```bash
git push -u origin HEAD
```

或当前已跟踪上游时：

```bash
git push
```

- 推送前确认分支合理（一般在 `main` 或用户指定的 `feature/*`）。
- **禁止**默认执行 `git push --force` / `--force-with-lease`；只有用户**明确要求**改写远程历史时才可，并先说明风险。
- 推送失败：贴出原文（权限、网络、非快进等）+ 通俗解释 + 可选下一步（如先 `git pull --rebase` 需用户同意后再做）。

### 6. 汇报结果

固定结构：

1. **结论**：存档是否成功、推送是否成功  
2. **提交**：短哈希 + 完整说明  
3. **远程**：`origin` 与分支名；可给仓库链接（若已知）  
4. **未纳入本次提交的文件**（如有）及原因  

## 与其它能力的边界

| 需求 | 用什么 |
|------|--------|
| 日常改代码后保存并推 GitHub 源码 | **本技能 `/git-save`** |
| 启动开发窗口 | `/run-app` |
| 重新打安装包到本地 `release/` | `/rebuild-app` |
| 把 `.exe` 挂到 GitHub Releases | 网页上传或 API；**不是**本技能 |

## 安全与协作红线

- 不删除用户未提交工作（不做 `git reset --hard` / `git clean -fd` 除非用户明确要求并确认）。
- 不修改 git 历史（rebase/amend 已推送提交）除非用户明确要求。
- 不把安装包 `release/*.exe` 塞进源码库（应走 Release 附件）。
- 重大架构/依赖变更仍须先方案后动手；本技能只负责**已经发生的文件改动**的存档推送。

## 沟通要求

- 全程简体中文；结论 → 原因 → 接下来怎么做。
- 工具调用的 `description` 使用中文。
- 如实反馈：失败不粉饰、不假装已推送成功。
