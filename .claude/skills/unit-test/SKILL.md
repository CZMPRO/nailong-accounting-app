# 单元测试（方案二：智能补测 + 执行 + 中文报告）

本技能在**本仓库**完成三件事：

1. **创建/更新**单元测试（检查表）  
2. **执行**测试  
3. **给出**零基础可读的中文测试报告  

默认**不**在本技能内直接拦截 Git 提交、不自动 `/git-save`、不改业务功能“为了让测试凑合通过”（发现业务 bug 先报告；修代码需符合项目协作规范）。

**通行证：** 仓库启用了 **Claude Code** 双通行证门禁时，由子代理 **`tester`** 在全部用例真实通过后写入 `.claude/states/tester-pass.json`（本技能可不直接写文件；若仅用技能而不经 tester，则不会自动发通行证）。失败时不得写 `passed`。

---

## 技术栈（固定）

| 项 | 约定 |
|----|------|
| 测试框架 | **Vitest** |
| 语言 | TypeScript |
| 环境 | 优先 `environment: 'node'`（测主进程/数据库/纯逻辑） |
| 用例文件 | `*.test.ts` 或 `*.spec.ts`，与被测文件同目录或就近 `__tests__` |
| 运行命令 | 见下方「如何执行」 |

当前仓库在回滚后可能**尚未**配置 `test` 脚本或项目内用例：本技能允许在**首次需要时**补齐最小 Vitest 脚手架，但**新增 npm 依赖前必须先向用户说明并征得同意**（全局协作：重大依赖变更方案确认）。

### 首次脚手架（仅当缺失且用户同意）

若 `package.json` 无 `vitest` / 无 `test` 脚本，且用户同意安装：

1. 安装：`npm install -D vitest`  
2. 在 `package.json` 的 `scripts` 增加：  
   - `"test": "vitest run"`  
   - `"test:watch": "vitest"`  
3. 如需独立配置，可增加简洁的 `vitest.config.ts`（`environment: 'node'`，`include` 指向项目内 `**/*.{test,spec}.ts`，**排除** `node_modules`）  
4. 若存在 Windows 盘符大小写导致 Vitest 找不到用例的历史问题，可使用/恢复 `run-tests.js` 包装；有则优先：  
   - 全部：`npm test` 或 `node run-tests.js run`  
   - 单文件：`node run-tests.js run electron/database.test.ts`  

用户**不同意**装依赖时：停止写依赖相关改动，只报告现状与可选方案。

## 测试范围（方案二默认）

### 优先（默认要做）

- `electron/database.ts`：分类种子、增删改查流水、月度统计、清空/出厂重置、迁移逻辑等  
- 其它**纯函数/可脱离窗口运行**的账本核心逻辑  
- 用户点名的模块  

### 默认不做（除非用户明确要求）

- React 页面点击流、Ant Design 组件观感  
- `TankGame` 手感/帧同步（可单列，不作为第一版默认）  
- 真实弹系统对话框的 CSV 导入导出全流程（可对纯解析函数单测）  
- 安装包 / Electron 窗口是否启动（那是 `/run-app`、`/rebuild-app`）  

### 怎么选“测什么”

1. 看 `git status` / `git diff`：本次改动的逻辑优先补测  
2. 无改动时：对 `database` 等核心做回归（已有测试则跑全量并视缺口补强）  
3. 一次会话保持范围克制：宁可少而稳，不要虚构测不到的 UI

## 执行步骤（严格顺序）

### 1. 摸底

- 确认在仓库根目录（`package.json` 的 `name` 为 `nailong-bookkeeping`）  
- 查看是否已有 Vitest、`npm test`、`run-tests.js`、已有 `*.test.ts`  
- 查看本次 git 改动与 `electron/database.ts` 等关键文件  
- 用中文向用户一句话说明：将测哪些模块  

### 2. 脚手架（按需）

- 缺 Vitest / test 脚本 → **先征得用户同意**再安装配置  
- 已具备 → 跳过  

### 3. 创建或更新测试

- 有对应 `*.test.ts`：按改动**更新/补充**用例，避免无意义重复  
- 无：新建，例如 `electron/database.test.ts`  
- 用例应包含：正常路径 + 1～2 个边界（空账本、错误 id、月份筛选等，按真实 API 设计）  
- 测试代码注释用**简体中文**；命名清晰（`describe` / `it` 可用中文描述字符串，便于报告阅读）  
- **不要**在测试里连接用户真实 `userData` 账本；使用临时目录或内存/可注入路径，避免污染真实记账数据  
- 若 `database` 强依赖 `app.getPath`，优先通过已有 `initDatabase(自定义路径)` 一类 API 指向临时文件夹；测完可清理临时文件  

### 4. 执行测试

优先顺序：

1. `npm test`（若已配置）  
2. `node run-tests.js run`（若存在包装）  
3. `npx vitest run`  
4. 单文件：`npx vitest run <路径>` 或 `node run-tests.js run <路径>`  

注意：

- 工具 `description` 用中文  
- 保留完整终端输出，供报告引用  
- Windows 下若遇盘符/`ELECTRON_RUN_AS_NODE` 类问题，记录在报告「环境说明」；单测应尽量不启动 Electron 窗口  

### 5. 输出中文测试报告（必须）

对用户回复使用固定结构：

```markdown
## 测试结论
- 通过 / 失败（一句话）

## 测了什么
- 模块与用例范围（大白话）

## 结果一览
- 总数 / 通过 / 失败 / 耗时（若日志有）

## 失败详情（如有）
- 哪个用例：期望 vs 实际
- 可能原因（通俗）
- 建议下一步

## 本次新增或修改的测试文件
- 路径列表

## 说明
- 未覆盖的范围（如 UI、小游戏）
- 是否动过 package.json / 依赖
```

- **失败时**：不伪装成功；不默认 `--no-verify` 去提交  
- **通过时**：可提示「若要存档可用 `/git-save`」，但**不自动**提交  

### 6. 失败后的修复策略

- 若是**测试写错**：改正测试，再跑一轮  
- 若是**产品代码疑似 bug**：用大白话说明，**先报告**；是否改业务代码遵循「方案确认 / 用户同意」，不大面积顺手重构  
- 同一轮最多自动「改测试并重跑」到结果稳定；避免无限循环

## 与其它技能的边界

| 技能 | 关系 |
|------|------|
| `/git-save` | 本技能**默认不调用**；测试挂了仍可单独存档（方案二不设门禁） |
| `/run-app` | 启动窗口，不代替单元测试 |
| `/rebuild-app` | 打包，不代替单元测试 |

用户若要求「提交前必须测试通过」，属于升级到更严门禁，需另改 `git-save` 或钩子——**本技能正文默认不做**。

---

## 沟通与安全

- 全程简体中文；结论 → 原因 → 下一步  
- 不删除用户真实账本数据  
- 不把密钥写进测试  
- 不强制推送、不 `reset --hard`  
- 新增依赖、改动数据库真实文件结构等重大项：先说明再执行  

## 成功标准

- 有可运行的 Vitest 用例覆盖约定范围内的核心逻辑（至少数据库相关或用户点名模块）  
- 已实际执行测试并引用真实输出  
- 用户能看懂的中文报告已给出
